核心机制

读路径采用 Read-Through 模式： 请求先查 L1 缓存，未命中则查 L2 缓存，仍未命中才 穿透 到数据库。

写路径采用 Write-Behind Invalidation： 数据库写入后发布变更事件，Invalidator 消费事件后 执行 延迟双删， 保证缓存与数据库的最终一致性。

为应对 缓存击穿，对热点 key 使用 Singleflight 机制： 同一 key 的并发回源请求合并为一次数据库查询。

术语表
Read-Through
缓存自动代理数据加载，调用方无需感知缓存未命中的处理逻辑
L1 缓存
进程内 LRU，容量有限但延迟极低（<0.1ms），每个实例独立
L2 缓存
Redis Cluster 共享缓存，容量大，延迟 ~2ms，所有实例共享
穿透
请求绕过所有缓存直达数据库，是缓存系统需要防护的场景
Write-Behind Invalidation
写操作后异步通知缓存失效，而非同步更新缓存值
延迟双删
写入后立即删缓存，延迟 500ms 再删一次，覆盖并发读重建的脏数据
缓存击穿
热点 key 过期瞬间大量请求同时穿透到数据库
Singleflight
同一 key 的并发请求合并为一次实际调用，避免重复回源

async get(key: string): Promise<Order | null> {
  // L1: 进程内 LRU
  const l1 = this.lru.get(key);
  if (l1) return l1;

  // L2: Redis (with singleflight)
  return this.singleflight.do(key, async () => {
    const l2 = await this.redis.get(key);
    if (l2) { this.lru.set(key, l2); return l2; }

    // 穿透: 查 DB + 回写
    const row = await this.db.findOrder(key);
    if (row) {
      await this.redis.set(key, row, { ttl: 300 });
      this.lru.set(key, row);
    }
    return row;
  });
}

Core Mechanisms

The read path uses a Read-Through pattern: requests first check the L1 cache, then the L2 cache on miss, and only penetrate to the database when both miss.

The write path uses Write-Behind Invalidation: after a database write, a change event is published; the Invalidator consumes the event and performs delayed double-delete, ensuring eventual consistency between cache and database.

To handle cache stampede, hot keys use Singleflight: concurrent backfill requests for the same key are merged into a single database query.

Glossary
Read-Through
Cache automatically proxies data loading; callers need not handle cache miss logic
L1 Cache
In-process LRU with limited capacity but extremely low latency (<0.1ms); each instance is independent
L2 Cache
Redis Cluster shared cache with large capacity and ~2ms latency; shared across all instances
Penetration
Requests bypass all caches and hit the database directly — a scenario cache systems must guard against
Write-Behind Invalidation
Asynchronously notifies cache invalidation after writes, rather than synchronously updating cache values
Delayed Double-Delete
Delete cache immediately after write, then delete again after 500ms to cover stale data rebuilt by concurrent reads
Cache Stampede
A burst of requests penetrating to the database when a hot key expires
Singleflight
Merge concurrent requests for the same key into a single actual call, avoiding duplicate backfills

async get(key: string): Promise<Order | null> {
  // L1: in-process LRU
  const l1 = this.lru.get(key);
  if (l1) return l1;

  // L2: Redis (with singleflight)
  return this.singleflight.do(key, async () => {
    const l2 = await this.redis.get(key);
    if (l2) { this.lru.set(key, l2); return l2; }

    // Penetration: query DB + write back
    const row = await this.db.findOrder(key);
    if (row) {
      await this.redis.set(key, row, { ttl: 300 });
      this.lru.set(key, row);
    }
    return row;
  });
}

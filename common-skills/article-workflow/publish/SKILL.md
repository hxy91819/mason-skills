# Article Workflow Publish

This is Phase 8 of the `article-workflow-*` series, corresponding to "Channel Publishing & Syncing" in the article optimization workflow.

Goal: Sync the author-maintained `06-final-review/final.md` into channel drafts. Image URLs are managed via `08-publish/image-map.json`; text remains fully consistent with the source. Scripts handle mechanical substitution; the skill invokes the LLM to find "adaptations needed beyond images, per-channel" and reports them (**report only, no modifications**).

## Trigger Scenarios

Use this skill when the user asks to:

- Sync `final.md` to internal wiki / WeChat Official Account / Zhihu, etc.
- Regenerate `08-publish/<channel>/<channel>-meta.md` and `<channel>-body.md`
- Run reverse extraction (first-time sync when the author exports an already-published draft with embedded images from the channel editor)
- Check sync status between channel drafts and the source
- Validate that `image-map.json` is complete
- Add a new channel (`wechat` / `zhihu`, etc.)

## Inputs & Outputs

Default inputs:

- `.article-workflow/06-final-review/final.md` (source)
- `.article-workflow/08-publish/image-map.json` (mapping)
- Optional: `.article-workflow/08-publish/<channel>/<channel>-published.md` (for first-time reverse extraction)

Default outputs:

- `.article-workflow/08-publish/<channel>/<channel>-meta.md` (title + lead paragraph; maintained independently for the internal wiki channel)
- `.article-workflow/08-publish/<channel>/<channel>-body.md` (body text, excluding title and lead)
- `.article-workflow/08-publish/<channel>/<channel>.lint.json` (machine-level lint output)
- Skill output: LLM report (printed in conversation, not written to a file)

Additional outputs on first-time image pasting (`sync --templated`):

- `.article-workflow/08-publish/<channel>/<channel>-templated.md` (full draft preserving `imgs/xxx` paths)
- `.article-workflow/08-publish/<channel>/<channel>-meta-templated.md` (title + lead, preserving `imgs/xxx`)
- `.article-workflow/08-publish/<channel>/<channel>-body-templated.md` (body text, preserving `imgs/xxx`)

If the user provides an article directory, look for `.article-workflow/` under that directory first, then `08-publish/`.

## Core Principles

- **Single source of truth**: `06-final-review/final.md` is the sole human-editable entry point. All channel drafts under `08-publish/` are generated by scripts.
- **Internal wiki channel meta is independent**: The title and lead are maintained independently in `<channel>-meta.md`; the body text is maintained in `<channel>-body.md`. When posting to the internal wiki, paste the title and lead from meta, and the body from body.
- **Image mapping uses channel as top-level key**: The top level of `image-map.json` is the channel name (`wiki` / `wechat` / `zhihu`), with `{source_image: URL}` as inner values.
- **Scripts only do mechanical substitution + machine lint**; other channel adaptation issues are reported by the LLM.
- **LLM reports are read-only, no modifications**: Protect the author's voice. The LLM lists "specific line numbers + suggested actions"; the author decides.
- **Wiki-published draft text must not be changed**: Altering text would break the alignment for reverse extraction. The skill performs a text-consistency diff before extraction and stops immediately if discrepancies are found.
- **Missing mapping defaults to error**: On the internal wiki channel, relative paths will break images; mappings must be added first. Use `--allow-missing` for placeholder mode.
- **Substitution pattern includes context**: `![](imgs/X)` or `src="imgs/X"`, not bare `imgs/X` (to prevent false substitutions of path mentions in body text).
- **When encountering ambiguities that would affect publishing decisions (first-time vs. routine, missing image handling, special rules for new channels), proactively ask the author**; if already clear, proceed directly and state key assumptions in the final reply.

## Workflow

### Step 1: Read Configuration & Source

Read:

1. `06-final-review/final.md` (source)
2. `08-publish/image-map.json` (mapping)
3. `08-publish/README.md` (workflow instructions)

Determine current state:

- Does `image-map.json` exist and is it non-empty?
- Did the author mention "first-time", "reverse extraction", or "extract"?
- Did the author mention "adding a new channel"?

### Step 2: Determine Mode and Confirm with Author

Based on the user's request and current state, determine which mode applies. **Use `AskUser` to confirm**, avoiding accidental runs:

- **Routine sync**: Source and mapping are ready; the author edited `final.md` and wants to regenerate channel drafts
- **First-time reverse extraction**: The author exports an already-published draft with images from the internal wiki as `wiki-published.md` and wants to write the mapping into `image-map.json`
- **Adding a new channel**: The author added a top-level key (new channel) to `image-map.json` but hasn't pasted images yet
- **Validation + LLM report**: Run lint + report, without modifying any files

If the user's request is already clear (e.g., "run sync"), skip this step.

### Step 3: Execute the Corresponding Sub-command

**Routine sync**:

```bash
bash 08-publish/sync-publish.sh sync [--channel CHAN]... [--allow-missing]
```

After running: check exit code, read `08-publish/<channel>/<channel>-meta.md` and `<channel>-body.md` to confirm they exist, and use `diff` to verify the text portion is fully consistent with `final.md` (image portions replaced per mapping).

**First-time reverse extraction**:

```bash
bash 08-publish/sync-publish.sh extract --channel CHAN --published <published-md>
```

After running:

- Check exit code (0 = success)
- Read `image-map.json` to confirm mapping was written
- Run `sync` to regenerate `<channel>-meta.md` and `<channel>-body.md`, confirming substitution is effective
- Check that text portions are consistent with `final.md`

**Adding a new channel**:

1. Prompt the author: edit `image-map.json` to add a top-level key
2. Run `sync --templated` to generate the templated draft
3. Prompt the author to paste images in the new channel's editor
4. Run `extract` to reverse-extract the mapping

**Validation + LLM report**:

```bash
bash 08-publish/sync-publish.sh validate
bash 08-publish/sync-publish.sh lint --channel CHAN --machine
```

### Step 4: Run LLM-Layer Lint

Machine-level lint has been run via `lint --machine`. The LLM layer is executed by this skill in the conversation:

1. Read `08-publish/<channel>/<channel>-body.md` and `06-final-review/final.md`
2. Identify channel adaptation issues the LLM should flag (that machine-level lint cannot catch):
   - Code blocks that may lose indentation during internal wiki rendering (especially inside nested lists)
   - Whether links can navigate within the channel
   - Visual weight of blockquotes after channel rendering
   - Whether tone carriers might be destroyed by the channel editor's "auto-optimization"
   - Emoji (🦞 🦀 🐚, etc.) rendering compatibility
3. Compile into a report, **listing only issues and suggestions, not modifying channel drafts**:

```
LLM Report: <channel>
- line N: <specific issue> → suggest <specific action>
- line M: <specific issue> → suggest <specific action>
...
```

4. If there are no issues, output `LLM Report: <channel> No issues`.

### Step 5: Validation

Post-completion checks:

- Is `08-publish/image-map.json` valid?
- Does `08-publish/<channel>/<channel>-meta.md` exist?
- Does `08-publish/<channel>/<channel>-body.md` exist?
- Is `08-publish/<channel>/<channel>-meta.md` content consistent with `final.md` title + lead?
- Is `08-publish/<channel>/<channel>-body.md` content consistent with `final.md` body (images replaced per mapping)?
- Does `08-publish/<channel>/<channel>.lint.json` exist?
- Has the LLM report been output?

## Author Decision Dialog

This stage has a clear "first-time vs. routine" mode choice that affects the execution path. Use `AskUser` to ask in one pass:

- Which sub-command to run? sync / extract / lint / validate
- Which channel? wiki / wechat / all

If the user's request is already clear (e.g., "sync wiki", "add wechat channel"), do not ask again.

## Output Style

- Use English unless the user specifies otherwise.
- The final reply only summarizes:
  - Which sub-command was run
  - Location of generated artifacts
  - Whether text consistency passed
  - List of issues from the LLM report
  - Remaining items for the author to decide (e.g., "new images not yet pasted to internal wiki")
- Do not output the full text; do not produce verbose publishing checklists.
- **Important reminder**: Channel drafts are generated by scripts. After the next `final.md` change, simply run `bash 08-publish/sync-publish.sh sync`; do not manually edit channel drafts.
